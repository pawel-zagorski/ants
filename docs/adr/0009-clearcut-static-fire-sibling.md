# Clearcut is a static, Drone-detected sibling of Fire, not an Event

We're adding **Illegal Deforestation** as a new detectable phenomenon: a **Clearcut** that occupies a fixed patch of forest, is spotted by patrolling Drones, and is inspected the way a Fire is. It shares a lot with Fire — a hex **Clearcut Footprint**, an Undetected → Detected → Investigated tier lifecycle, a from-a-distance **Uncertainty Ellipse** estimate, and (later, issue Y) an orbit-revealed **Confirmed Shape** — but it differs on the two axes that made Fire its own concept in ADR-0004: it is **static** (no wind term, no time/spread term; the footprint cell set is constant for the whole run) and it is **Drone-only** to detect (a Tower never detects a Clearcut, unlike a Fire, which is Tower-detected).

We modeled Clearcut as its **own top-level concept, a sibling of `Fire`** (`ClearcutRuntimeState`, `SimulationState.clearcuts`), rather than as a new `EventType` case, following the ADR-0004 pattern that split Fire out of the `Event` union. The alternative — adding a `Clearcut` case to `EventType` — was rejected for the same reasons Fire was: Person Sighting and Fallen Tree stay small, static, three-state Events, uncontaminated by footprint/tier/estimate fields that would always be undefined for them, and no future engineer has to reason about a Clearcut's oriented-ellipse footprint or its `detected` tier when touching plain Events. Keeping it a separate map alongside `events`/`fires` also keeps detection asymmetries honest: the drone-only rule lives in the Clearcut branch of `advanceSimulation` and can't accidentally leak into Event/Fire detection.

We deliberately **reuse Fire's machinery** rather than reinventing it: the hex-grid/growth-ellipse math computes the footprint (as an oriented ellipse with the wind and time terms dropped — see `engine/clearcutFootprint.ts`), and the same `uncertaintyEllipseRadiusMeters` sizes the estimate, called with the time-growth term hard-zeroed (`uncertaintyEllipseRadiusMetersForClearcut`) since a Clearcut never spreads. The estimate is sized by the detecting Drone's distance **at the moment of detection** (recorded sticky on the runtime state), not a live per-tick distance, so the fuzzy estimate stays fixed once seen. Detection is deterministic/sticky by distance per ADR-0003, and payload does not gate it (any Drone qualifies). The Scenario authors a Clearcut in the same timed entry list as Events and Fire ignitions (a `Clearcut` entry with `position` + `semiMajorAxisMeters`/`semiMinorAxisMeters`/`orientationDegrees` + `spawnAtSimSeconds`, no `durationSimSeconds` since it never auto-resolves), purely to avoid schema churn — `partitionScenarioEntries` routes it into `clearcuts`.

See `CONTEXT.md` for the glossary terms this introduced or updated (Clearcut, Clearcut Footprint, the updated Uncertainty Ellipse, Patrol Route). Investigation/dispatch (Bingo-Range only, no One-Way Mission) and the discovered-on-site Person Sighting payoff are out of scope for this slice and land in issue Y.
