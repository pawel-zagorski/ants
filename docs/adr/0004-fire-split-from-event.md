# Fire is a distinct domain concept, not an Event

Fire used to be one case in the `Event` union (`Fire | PersonSighting | FallenTree`), all sharing a single static-point, Undetected → Detected → Resolved lifecycle. We're introducing continuous, wind-biased spread across a hex-grid Fire Footprint (via a deterministic Growth Ellipse model, chosen over a cellular-automaton spread to keep ADR-0003's reproducibility without per-tick neighbor bookkeeping), plus a distinct Undetected → Tower-Detected → Investigated tier system — neither of which fits "static in position" or the old three-state lifecycle.

We picked splitting Fire into its own concept over bolting optional fields onto `Event`, so Person Sighting and Fallen Tree keep their simple, static definition uncontaminated by Fire's much richer state, and so a future engineer doesn't have to reason about spread/wind/hex-grid fields that are always undefined for two out of three cases. Fire still shares a Scenario's single timed authoring list with Events (an ignition entry, not a separate list) purely to avoid schema churn — see `CONTEXT.md` for the full set of new terms this introduced (Fire, Wind, Fire Footprint, Growth Ellipse, Uncertainty Ellipse, Confirmed Shape).
