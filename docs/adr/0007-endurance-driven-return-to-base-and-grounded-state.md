# Endurance-driven return-to-base and a Grounded state

ADR-0006 scoped real endurance gating to Fire dispatch only and explicitly deferred "a fully general patrol-drain/return-to-base/recharge simulation" as a separate future effort, leaving ordinary patrol endurance cosmetic. This decision partially reverses that deferral: any active Drone (patrolling, investigating an Event, or on a round-trip Fire investigation — but never a One-Way Mission Drone, which is intentionally sacrificial, nor an already-Lost/Grounded one) now continuously checks whether its remaining flight budget (`remainingEnduranceSimSeconds × cruiseSpeedMetersPerSecond`) still covers the straight-line distance to its *nearest* Base Station. The instant it does not, the Drone abandons whatever it was doing, flies to that nearest Base Station, and enters a new terminal **Grounded** state — parked at the base, permanently out of service for the run and excluded from dispatch.

We deliberately stopped short of the full lifecycle: there is **no recharge and no relaunch**. Because endurance remains the closed-form `max(0, maxEnduranceSimSeconds − elapsedSimSeconds)` drain from t=0 (unchanged from ADR-0006), a Grounded Drone stays Grounded for the rest of the run rather than resetting and returning to patrol. Recharge would require replacing that closed-form model with stateful, resettable per-Drone endurance — a much larger change that re-opens ADR-0006 wholesale and is again deferred. Grounded is kept distinct from **Lost**: Lost means sacrificed in the field on a One-Way Mission (frozen wherever it ran out); Grounded means it made it safely home but is depleted. The threshold uses no safety reserve, faithful to the aviation "bingo" meaning ("just enough to still make it back") and to the existing `bingoRange.ts` distance/speed math.
