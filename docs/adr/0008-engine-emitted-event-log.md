# The Event Log is emitted by the engine, not diffed from snapshots

The Event Log — the bottom-of-page textual feed of what is happening — needs to reliably capture transitions the operator cares about, including "reached a Fire and began orbiting" and "reached bingo endurance, returning to base." The engine is a pure, deterministic snapshot machine (ADR-0003) whose `activityAfterInvestigationExpiry` loops until stable, so at high speed or a 30s step a Drone can pass through `travelingToFire → investigatingFire → returningToBase → patrolling` entirely within one tick. Any approach that merely diffs consecutive React snapshots would therefore *miss* those collapsed intermediate transitions and could only attribute frame-time (not exact) timestamps.

We chose to make `advanceSimulation` and the manual-command functions (`withManualDispatch`, `withManualFireDispatch`, and the "Return to Nearest Base" recall `withManualReturnToBase`) emit an append-only log of structured entries carried inside `SimulationState`, each stamped with the exact sim-second the transition occurred. This trades a departure from the "state is pure current-snapshot only" shape (the log is history that grows, and it is capped at ~1000 entries, dropping oldest) for completeness and determinism: every logged transition is recorded exactly once with an accurate simulated timestamp, and the log resets naturally with the rest of state when the World/Scenario changes. Entries store typed payloads (ids, positions, severity), not pre-rendered strings — the human-readable line and its Scenario-Epoch timestamp are formatted in the UI layer, keeping wording and formatting out of the pure engine. The log is operator-facing (fog-of-war): it records detections, dispatches, and observed Drone activity, never Undetected ground-truth spawns, and the Ground Truth View does not add lines to it.
