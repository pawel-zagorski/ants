# Forest Situational Awareness Simulation

## Problem Statement

An operator wants to understand what it would feel like to monitor a forest for fires, intruders, and hazards using a network of fixed towers and patrolling drones — without needing real hardware, a real incident, or a real backend deployment to see it in action. Today there's no way to visualize how such a sensor network would actually notice things happening in a forest, how quickly, and from which asset, or to compare different event sequences (a quiet day vs. a wildfire outbreak) against the same fixed sensor layout in a repeatable way. Nor is there any way to see how uncertain early detections really are, how a wildfire would actually grow and spread given the wind, or to practice the operator's own judgment call of which drone to risk sending to investigate it.

## Solution

A web-based, frontend-only dashboard that renders a real 50×50km area of Rokua National Park, Finland on a real map, shows the fixed sensor network (Towers, Base Stations, Drones) as clickable assets with live status, and plays back a chosen, reproducible Scenario of spawned Events (person sightings, fallen trees) and Fires. The operator watches Drones patrol and Towers stand watch, sees Events and Fires appear on a "ground truth" layer only if they opt in, and otherwise only sees what the network has actually Detected — with play/pause/speed controls over the Simulation Clock. A Fire is never a simple static point: once ignited it spreads continuously across a hex-grid footprint, biased downwind by the Scenario's Wind, and the operator only ever sees an Uncertainty Ellipse standing in for its real size/position until they choose to risk a Drone flying out to map its Confirmed Shape. Every Detection — Event or Fire — now requires the operator to click it and manually choose which Drone to send from a status panel; for a Fire, that panel breaks eligible Drones into those that can safely round-trip (Bingo Range) and those that can only reach it as a One-Way Mission the Drone won't survive. Because a Scenario is just a static, timed ignition/event script plus a fixed Wind, replayed deterministically against a static World, the exact same run can be reproduced or shared as a JSON file.

## User Stories

### Map & Assets

1. As an operator, I want to see a real map of a 50×50km area of Rokua National Park so that the simulation feels grounded in a real place, not an abstract grid.
2. As an operator, I want to pan and zoom the map so that I can inspect specific areas closely or see the whole monitored region at once.
3. As an operator, I want to see Tower icons at their real/plausible positions on the map so that I understand the fixed long-range fire-watch coverage.
4. As an operator, I want to click a Tower to see its status (operational state, detection radius, and any Fire it's currently tracking) so that I can assess its current relevance.
5. As an operator, I want to see Base Station icons on the map so that I know where Drones live and return to recharge.
6. As an operator, I want to click a Base Station to see which Drones are docked there, which are out on patrol/investigation, and its operational status, so that I understand its current load.
7. As an operator, I want to see Drone icons on the map that move in real time (at the current Simulation Clock speed) so that the patrol behavior feels alive rather than static.
8. As an operator, I want Quadrocopters and Fixed-Wing Drones to be visually distinguishable so that I can tell at a glance which kind of asset I'm looking at.
9. As an operator, I want Quadrocopters to patrol a tight loop near their Base Station so that their short-range role is visually obvious.
10. As an operator, I want Fixed-Wing Drones to patrol a long perimeter loop at higher speed so that their long-range role is visually obvious.
11. As an operator, I want to click a Drone to see rich telemetry (state, battery %, position, speed, heading, assigned Event/Fire if any, remaining flight endurance) so that I can assess its condition and current task the way a real operator would.
12. As an operator, I want a dropdown/start screen listing bundled Scenarios (e.g. "Quiet Day", "Wildfire Outbreak", "Hiker Intrusion") so that I can pick a specific story to watch without editing files.
13. As an operator, I want to press Play to start the chosen Scenario so that events and fires begin spawning and the sensor network begins reacting.
14. As an operator, I want to pause the Simulation Clock at any point so that I can inspect the current state without it changing.
15. As an operator, I want to change the Simulation Clock's speed multiplier (e.g. 1x/10x/60x) so that I can watch slow stretches quickly and interesting moments more closely.
16. As an operator, I want to step the Simulation Clock forward by a fixed increment while paused so that I can advance frame-by-frame through an interesting moment.
17. As an operator, by default I want to see only Events/Fires the sensor network has actually Detected (fog of war) so that the dashboard reflects genuine situational awareness, not omniscience.
18. As an operator, I want a toggle to reveal Ground Truth (all Events, all Fires' real footprints, including undetected ones shown faded/dashed) so that I can see how much the network is missing or how uncertain its picture currently is.

### Detection & Manual Dispatch

19. As an operator, I want a Fire to only be detectable by a Tower (not a Drone that hasn't happened to fly over it) unless a Drone is also within range, so that the different sensor roles feel meaningfully different.
20. As an operator, I want Person Sighting and Fallen Tree Events to only be detectable by Drones (never by Towers) so that Tower and Drone roles remain distinct.
21. As an operator, when a Tower or Drone Detects a new Event or Fire, I want to click it and manually choose which Drone to send to investigate, rather than the network auto-dispatching on its own, so that I stay in control of every decision to commit a Drone.
22. As an operator, for a Person Sighting or Fallen Tree, I want a simple list of currently-available Drones to pick from and send, so that dispatching a routine, low-stakes Event is quick.
23. As an operator, for a Fire, I want the panel to separately list Drones that can reach it and still safely return to a Base Station (Bingo Range) from Drones that can only reach it as a One-Way Mission I know will lose the Drone, so that I can make an informed, deliberate risk decision proportional to how urgent the fire is.
24. As an operator, I want an investigating Quadrocopter to hover directly over a Person Sighting/Fallen Tree it's investigating, and an investigating Fixed-Wing Drone to circle it instead, so that the physical constraints of each Drone type are respected.
25. As an operator, once a Drone finishes investigating a Person Sighting/Fallen Tree, I want it to resume its normal patrol loop so that the simulation doesn't get stuck in a permanent "investigating" state.
26. As an operator, I want a Detected Person Sighting/Fallen Tree to visually change (e.g. to "resolved"/greyed-out) after the Scenario's configured duration for it elapses, so that old events don't clutter the map indefinitely while still leaving a visible history. (Fires never auto-resolve this way — see Fire Behavior below.)

### Fire Behavior

27. As an operator, when a Fire is only Tower-Detected, I want to see an Uncertainty Ellipse standing in for its real position/size (sized by the detecting Tower's distance and growing the longer it goes uninvestigated) instead of a precise marker, so that the dashboard honestly represents how little a distant Tower sighting actually tells you.
28. As an operator, I want a Fire to spread continuously across a hex-grid footprint once ignited, growing fastest downwind and slowest upwind based on the Scenario's Wind, so that watching a Fire feels like watching a real wildfire rather than a static icon.
29. As an operator, I want to see the current Wind direction/speed on the map so that I can anticipate which way a Fire is likely to grow.
30. As an operator, I want to click a Fire to see its Detection Status (which asset detected it, when, and its current tier — Tower-Detected vs. Investigated), so that I understand how much is actually known about it before deciding whether to commit a Drone.
31. As an operator, once I send a Drone to investigate a Fire, I want it to orbit the fire's current extent (rather than hover at a single point or circle a fixed small radius) so that it can physically map the fire's shape rather than just confirm a point.
32. As an operator, while a Drone is actively orbiting a Fire, I want the map to show its live Confirmed Shape (updating in real time) so that I can watch the true footprint reveal itself as the Drone flies.
33. As an operator, once an investigating Drone completes one full orbit lap of a Fire, I want a Bingo-Range Drone to automatically head back to a Base Station, and the Confirmed Shape to freeze as a stale snapshot from that moment, so that I understand my knowledge of the fire is only as fresh as its last overflight.
34. As an operator, when I knowingly send a Drone on a One-Way Mission to a Fire, I want it to keep orbiting and mapping for as long as its remaining endurance lasts, then become permanently Lost (frozen in place, greyed out, no longer dispatchable) the instant it runs out, so that the risk I accepted plays out visibly rather than being hand-waved away.

### Scenario & Reproducibility

35. As an operator, I want loading the same Scenario file twice to produce the exact same sequence of Detections, Fire growth, and Drone movements, so that I can trust the simulation for demoing or debugging.
36. As an operator, I want the World (map bounds, Towers, Base Stations, Drones) to be defined once in `world.json` and referenced by every Scenario, so that switching Scenarios never silently changes the sensor network layout.
37. As a scenario author, I want to write a new `scenario-*.json` file containing a single timed list of Events and Fire ignitions (type, position, spawn time, optional duration for Events) plus a fixed Wind and a calendar start date/time (Scenario Epoch), so that I can create new test scenarios without touching the World or engine code.
38. As an operator, I want every timestamp shown in the UI (detections, telemetry, the Simulation Clock itself) displayed as a real calendar date/time derived from the Scenario's Epoch, rather than raw elapsed seconds, so that the simulation reads like a real incident timeline.
39. As an operator, I want the default World to include 2 Towers, 2 Base Stations, and 4 Drones (a mix of Quadrocopters and Fixed-Wing Drones) so that there's enough going on to observe interesting handoffs without being overwhelming.

## Implementation Decisions

- **Stack**: React + TypeScript + Vite. Map rendered via `react-leaflet` with OpenStreetMap raster tiles (real geography, real zoom/pan), per ADR-0001 (frontend-only architecture — no backend, no server-pushed state; the whole Simulation Clock and engine run in-browser).
- **Geography**: Map centered on Rokua National Park, Finland (64.5644°N, 26.4947°E), a real former fire-lookout-tower site, with a 50×50km default viewport.
- **World / Scenario file split** (ADR-0002): `world.json` holds the fixed roster — map bounds, Towers (position, detection radius), Base Stations (position, docked Drones), Drones (id, type: Quadrocopter | FixedWingDrone, home Base Station, patrol parameters). `scenario-*.json` files hold a single timed list of Events (`{type: PersonSighting | FallenTree, position, spawnAtSimSeconds, durationSimSeconds?}`) and Fire ignitions (`{type: Fire, position, spawnAtSimSeconds}`, no duration), plus a fixed `wind` (direction + speed) and a `startDateTimeIso` Scenario Epoch. Scenarios reference/assume a World; they never redefine the roster.
- **Simulation engine**: A single pure function, `advanceSimulation(state, elapsedSimSeconds) -> newState`, owns all business logic: Drone movement along patrol loops, Drone dispatch/investigate/return/Lost transitions, Event/Fire spawning from the loaded Scenario, Detection checks, Event lifecycle transitions (Undetected → Detected → Resolved), and Fire lifecycle transitions (Undetected → Tower-Detected → Investigated). This function has no dependency on React, Leaflet, or wall-clock time — it is driven purely by `elapsedSimSeconds`, which the UI layer derives from the Simulation Clock's play/pause/speed/step state.
- **Detection model** (ADR-0003): Deterministic and distance-based only. An Event/Fire is Detected the instant an eligible, active asset's position is within that asset's fixed detection radius of it — no probability, no seeded RNG. Towers detect Fires only. Drones (either type) detect any Event type or a Fire.
- **Fire is its own domain concept, not an Event** (ADR-0004): Fire shares a Scenario's single timed authoring list with Events but is modeled separately because it grows continuously rather than sitting static. Its real, ground-truth extent (**Fire Footprint**) is a set of 50m hex cells in a local axial grid centered on its ignition point, grown deterministically by the **Growth Ellipse** model — an ellipse anchored at the ignition point, growing fastest at its downwind head, slowest at its upwind back, intermediate on the flanks, with eccentricity set by Wind speed (head rate of spread ≈ 0.1× wind speed).
- **Wind**: A fixed direction and speed authored once per Scenario (not on World), constant for the whole run — kept scenario-fixed and hand-authored, never randomized or time-varying, to preserve ADR-0003's reproducibility guarantee.
- **Fire fog-of-war tiers**: Undetected (invisible in default view; real Fire Footprint shown faded in Ground Truth View) → Tower-Detected (default view shows only an **Uncertainty Ellipse**, sized by the detecting Tower's distance and growing with elapsed time since Detection) → Investigated (default view shows the **Confirmed Shape** — the real Fire Footprint as last observed by an investigating Drone, live-updating while a Drone actively orbits, then freezing stale the instant that Drone leaves or is Lost). Ground Truth View always shows the real, live Fire Footprint regardless of tier.
- **Manual dispatch replaces auto-dispatch for every Event/Fire type** (ADR-0005): the nearest-available-Drone auto-dispatch on Detection is retired. The operator must click a Detected Event/Fire and choose a Drone from its status panel. Person Sighting/Fallen Tree get a simple panel (list of currently-available Drones). Fire gets a richer panel split into **Bingo Range** Drones (can round-trip: reach the fire, complete one orbit lap, and still reach a Base Station) and **One-Way Mission** Drones (can physically reach the fire but not return; the operator knowingly accepts the Drone will be Lost).
- **Endurance becomes a real gate, scoped to Fire dispatch only** (ADR-0006): `remainingEnduranceSimSeconds` now determines Bingo Range/One-Way Mission eligibility (reusing the Return Envelope's distance/speed math) and drives the terminal **Lost** state for a One-Way Mission Drone the instant its endurance hits zero. Ordinary patrol and Person Sighting/Fallen Tree dispatch remain unaffected — endurance stays cosmetic display there. A fully general patrol-drain/return-to-base/recharge simulation is explicitly deferred (see Out of Scope).
- **Drone investigation behavior**: For Person Sighting/Fallen Tree, unchanged — Quadrocopters hover directly over the point, Fixed-Wing Drones circle it at a fixed radius, for a fixed duration, then return to patrol. For a Fire, both Drone types orbit its current extent (radius scaled to the Fire Footprint/Uncertainty Ellipse's current size) instead of hovering/fixed-radius-circling; a Bingo Range Drone completes one full orbit lap then returns to patrol; a One-Way Mission Drone keeps orbiting until its endurance is exhausted, then becomes Lost in place.
- **Scenario Epoch**: Each Scenario authors a real calendar `startDateTimeIso` that `elapsedSimSeconds = 0` corresponds to. Every timestamp shown anywhere in the UI (Simulation Clock, Asset panels, Fire/Event Detection Status) is Scenario Epoch + elapsed Simulation Clock time, displayed as calendar date/time rather than raw seconds.
- **Visibility modes**: Default view shows Detected Events, each Fire's Uncertainty Ellipse/Confirmed Shape per its tier ("fog of war"). Ground Truth View additionally renders Undetected Events and every Fire's real, live Fire Footprint regardless of tier.
- **Asset status panels**: Clicking a Tower/Base Station/Drone opens a status panel, unchanged from before. Clicking a Fire or Event opens a status panel showing Detection Status (source, time, current tier) plus the appropriate dispatch section (simple list, or Bingo Range/One-Way Mission split for Fire).
- **Scenario selection**: A start screen/dropdown lists bundled example Scenarios ("Quiet Day", "Wildfire Outbreak", "Hiker Intrusion"); the operator selects one, then presses Play to begin.
- **Simulation Clock**: Adjustable speed multiplier (e.g. 1x/10x/60x) with play/pause/step controls, decoupled from wall-clock time.
- **Domain vocabulary**: Follows `CONTEXT.md` exactly — World, Scenario, Scenario Epoch, Simulation Clock, Tower, Base Station, Drone/Quadrocopter/Fixed-Wing Drone, Event, Fire, Wind, Fire Footprint, Growth Ellipse, Uncertainty Ellipse, Confirmed Shape, Bingo Range, One-Way Mission, Lost, Detection, Ground Truth View. Avoid the synonyms listed under `_Avoid_` in that file (e.g. never "Fire Event", never "bingo fuel", never "god mode" for Ground Truth View).

## Testing Decisions

- Concentrate tests on the pure `advanceSimulation(state, elapsedSimSeconds)` engine function — this is the seam that carries all business logic (movement, dispatch, Detection, Event/Fire lifecycle, Fire growth) and is fully decoupled from React/Leaflet, so it can be tested with plain unit tests and fixed `world.json` + `scenario-*.json` fixtures, no browser/DOM needed.
- Test only externally observable behavior of the engine: given a World + Scenario + elapsed simulated time, assert on the resulting Event/Fire states, which asset performed each Detection, Drone states/positions, and a Fire's Fire Footprint/Uncertainty Ellipse/Confirmed Shape — not internal implementation details of how the loop math is structured.
- Cover at minimum: a Tower detecting a Fire at the edge and outside of its radius; a Drone detecting a Person Sighting/Fallen Tree while patrolling; a Quadrocopter hovering vs. a Fixed-Wing Drone circling while investigating a Person Sighting/Fallen Tree; a Detected Event auto-resolving after its configured duration; the Growth Ellipse producing a wind-biased, deterministic Fire Footprint (same Wind + elapsed time → byte-identical hex cells); the Uncertainty Ellipse's size as a function of Tower distance and elapsed time since Detection; correct Bingo Range vs. One-Way Mission vs. unreachable classification of Drones for a given Fire and remaining endurance; a Bingo Range Drone completing one orbit lap and returning to patrol with a frozen Confirmed Shape snapshot; a One-Way Mission Drone orbiting until endurance hits zero and transitioning to Lost in place; and running the same World+Scenario twice producing byte-identical resulting state (reproducibility).
- No prior art for tests exists yet in this repo (greenfield project) — this test suite established the pattern of "engine tests are plain unit tests against pure functions; UI is not unit-tested for this prototype" for future scenario/engine work to follow.

## Out of Scope

- No backend service, database, or multi-user/real-time-sync support (see ADR-0001). Revisit only if this moves beyond a prototype.
- No probabilistic/missed detection, or sensor unreliability (see ADR-0003). Wind is the only weather-like factor modeled — no rain, humidity, or temperature.
- No general drone-energy simulation: endurance draining during ordinary patrol, autonomous return-to-base at a low-fuel threshold, and Base Station recharge behavior are explicitly deferred as a separate future effort (see ADR-0006). Endurance only gates the Fire Bingo Range/One-Way Mission decision and Lost-state termination for this version.
- No multi-Fire interaction: each Fire grows in its own local hex grid and Fires never merge, even if their footprints would geometrically touch.
- No wind changes within a single Scenario run (no gusts, no scripted shifts) — Wind is a single fixed value per Scenario.
- No scenario editor UI — new Scenarios are authored by hand-writing `scenario-*.json` files.
- No authentication, persistence of past runs, or analytics/reporting on simulation history.
- No support for changing the World roster (adding/removing Towers, Base Stations, or Drones) at runtime through the UI.

## Further Notes

- Real-world caveat: Finland decommissioned its dedicated fire-lookout tower network in the 1980s in favor of aerial patrol. The Tower positions here are anchored to a real, fire-historically-significant site (Rokua National Park's former Pookivaara fire tower) but represent a fictional modernized sensor network, not an active real-world system.
- `CONTEXT.md` and `docs/adr/0001`–`0003` were produced in the design session preceding the original version of this PRD and should be treated as authoritative for terminology and the architectural decisions referenced above. `docs/adr/0004`–`0006` were produced in a follow-up design session that introduced Fire growth, manual dispatch, and scoped endurance gating, reversing specific lines from the original version of this PRD's Out of Scope section — treat those ADRs as the record of *why* that reversal happened.
- Suggested three bundled example Scenarios to build first: "Quiet Day" (a couple of harmless Fallen Tree events, no Fire), "Wildfire Outbreak" (a Fire detected by a Tower, growing downwind, available for manual Drone dispatch), and "Hiker Intrusion" (a Person Sighting only detectable once a patrolling Drone happens to pass nearby, demonstrating the fog-of-war/Ground Truth distinction well).
